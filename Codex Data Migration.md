# Codex Data Migration Guide

English | [Simplified Chinese](Codex%20Data%20Migration.zh.md)

Strict version: Codex data on this machine mainly lives in three areas.

1. Core data: `C:\Users\Administrator\.codex`
   This includes skills, plugins, sessions, archived sessions, chat indexes, SQLite history databases, configuration, automations, memories, and related state.

2. Project and draft workspace: `C:\Users\Administrator\Documents\Codex`
   This includes files, drafts, work directories, and projectless-chat artifacts generated by Codex.

3. Desktop app state: `C:\Users\Administrator\AppData\Roaming\Codex` and `C:\Users\Administrator\AppData\Local\Codex`
   These are mostly Electron cache and UI state. They are not always required, but back them up too if you want the restored installation to be as close as possible to the old one.

## Safest Migration Principle

Fully quit Codex on the old computer before copying. Do not copy while Codex is running, because local chat history may include SQLite/WAL files such as `logs_2.sqlite`, `logs_2.sqlite-wal`, and `state_5.sqlite-wal`; copying them while active may produce an inconsistent backup.

Run this in PowerShell on the old computer:

```powershell
Get-Process | Where-Object { $_.ProcessName -like '*Codex*' }
```

If any Codex process exists, exit from the UI first, or at least confirm you are not writing to an active conversation.

## Backup Commands On The Old Computer

Replace `E:` with the drive letter of your external disk.

Backup command block:

    $Stamp = Get-Date -Format yyyyMMdd-HHmmss
    $BackupRoot = Join-Path 'E:\' CodexMigration-$Stamp
    New-Item -ItemType Directory -Path $BackupRoot -Force | Out-Null

    robocopy (Join-Path $env:USERPROFILE '.codex') (Join-Path $BackupRoot '.codex') /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

    robocopy (Join-Path $env:USERPROFILE 'Documents\Codex') (Join-Path $BackupRoot 'Documents\Codex') /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

    robocopy (Join-Path $env:APPDATA 'Codex') (Join-Path $BackupRoot 'AppData\Roaming\Codex') /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

    robocopy (Join-Path $env:LOCALAPPDATA 'Codex') (Join-Path $BackupRoot 'AppData\Local\Codex') /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

Robocopy return codes `0-7` usually mean success or acceptable differences. Treat `8` or higher as failure.

## Restore Steps On The New Computer

1. Install Codex on the new computer.
2. Launch it once and confirm it can run.
3. Fully quit Codex.
4. Connect the external backup drive.
5. Run the restore commands.

Restore command block:

    $BackupRoot = 'E:\CodexMigration-your-backup-folder'
    $Stamp = Get-Date -Format yyyyMMdd-HHmmss

    $CodexHome = Join-Path $env:USERPROFILE '.codex'
    if (Test-Path $CodexHome) {
        Rename-Item -LiteralPath $CodexHome -NewName .codex.new-install-$Stamp
    }

    robocopy (Join-Path $BackupRoot '.codex') $CodexHome /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

    $CodexDocuments = Join-Path $env:USERPROFILE 'Documents\Codex'
    New-Item -ItemType Directory -Path $CodexDocuments -Force | Out-Null
    robocopy (Join-Path $BackupRoot 'Documents\Codex') $CodexDocuments /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

    robocopy (Join-Path $BackupRoot 'AppData\Roaming\Codex') (Join-Path $env:APPDATA 'Codex') /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

    robocopy (Join-Path $BackupRoot 'AppData\Local\Codex') (Join-Path $env:LOCALAPPDATA 'Codex') /E /COPY:DAT /DCOPY:DAT /R:2 /W:2 /XJ

## Security Notes

`auth.json`, `.sandbox-secrets`, and Electron Local Storage under AppData may contain login state or sensitive material. If the backup disk is not encrypted, do not casually place the full backup in a cloud drive.

A more conservative approach is to migrate chats, skills, and configuration first, then sign in again on the new computer instead of keeping a long-term unencrypted full `.codex` backup.

Also note that chat records can migrate, but external project files referenced by those chats do not migrate automatically. For example, a path such as `F:\Project 1\...` is outside `Documents\Codex`; you must separately copy the corresponding project drive, Git repositories, virtual environments, Gurobi/license files, Node/Python dependencies, and data folders.

## Acceptance Checklist

After migration, open Codex and check:

1. Old chats are visible.
2. Custom skills are still present, for example personal skills under `.codex\skills`.
3. Plugin skills such as Brooks Lint are still present under `.codex\plugins`.
4. Opening an old conversation loads its content correctly.
5. If file links inside conversations are broken, copy the corresponding external project directories or correct the paths.

## Most Important Final Step

After Codex can start successfully on the new computer, run the following prompt in Codex and let it finish the cleanup itself:

```text
Act as a Codex configuration migration and repair expert. Audit and repair migration leftovers in the current Codex configuration on this computer.

Background:
I migrated Codex data from another Windows computer. The old username may be <OLD_USER>, and the current username is <NEW_USER>. The current Codex home is usually C:\Users\<NEW_USER>\.codex. Do not assume the path is correct. First confirm with whoami, $env:USERPROFILE, $env:CODEX_HOME, or actual file locations.

Goals:
1. Find and repair stale old usernames or old paths in Codex configuration, automations, skills, plugins, marketplace, sandbox, capability, SQLite, global state, and session indexes.
2. Preserve chat records, skills, automations, plugins, and personal configuration. Do not delete them crudely.
3. After repair, make sure Codex Desktop can normally read projectless chats, pinned chats, project chats, automations, and skills.

Mandatory safety rules:
- Before any write operation, create a backup on the current user's Desktop, for example C:\Users\<NEW_USER>\Desktop\codex-migration-backup-YYYYMMDD-HHMM.
- Do not blindly rewrite sessions/*.jsonl globally.
- If rollout JSONL must be modified, preserve these invariants:
  - The first byte of the file must not be a UTF-8 BOM.
  - The first line must be valid JSON.
  - The first event type must be session_meta.
  - payload.id must match the thread id in the filename.
  - Do not append records from another thread.
- Do not write JSONL back with PowerShell's default UTF-8 behavior. Use UTF8Encoding(false) or a structured JSON tool to write UTF-8 without BOM.
- Read and write .codex-global-state.json with a JSON parser. Do not cut strings manually.
- Before modifying SQLite, run .backup or copy the database file. After modification, run checkpoint/vacuum.

Inspection scope:
- C:\Users\<NEW_USER>\.codex\config.toml
- C:\Users\<NEW_USER>\.codex\cap_sid
- C:\Users\<NEW_USER>\.codex\.codex-global-state.json
- C:\Users\<NEW_USER>\.codex\automations
- C:\Users\<NEW_USER>\.codex\skills
- C:\Users\<NEW_USER>\.codex\plugins
- C:\Users\<NEW_USER>\.codex\.sandbox
- C:\Users\<NEW_USER>\.codex\.sandbox-bin
- C:\Users\<NEW_USER>\.codex\.sandbox-secrets
- C:\Users\<NEW_USER>\.codex\sessions
- C:\Users\<NEW_USER>\.codex\state_*.sqlite
- C:\Users\<NEW_USER>\.codex\logs_*.sqlite
- C:\Users\<NEW_USER>\.codex\sqlite\*.db
- C:\Users\<NEW_USER>\Documents\Codex

Repair tasks:
- Replace old paths C:\Users\<OLD_USER>, \\?\C:\Users\<OLD_USER>, C:/Users/<OLD_USER>, case variants, and JSON-escaped variants with the current user path.
- Repair marketplaces.*.source in config.toml. The personal marketplace source must not keep pointing to the old user directory.
- Repair automation cwds/source_cwd values.
- Repair old cwd and writable root mappings in cap_sid. If uncertain, clear path mappings and let Codex regenerate them.
- Clear stale DPAPI or old capability state in sandbox secrets/runner/cache, but do not delete normal user data.
- Repair projectless-thread-ids, pinned-thread-ids, thread-workspace-root-hints, electron-saved-workspace-roots, and project-order in .codex-global-state.json.
- Remove old workspace roots that no longer exist.
- Check all rollout JSONL files for no BOM, correct first session_meta event, and no obvious cross-thread concatenation.
- If cross-thread concatenation is found, cut only the tail that clearly belongs to another thread or another dated task, and keep a backup.

Final acceptance output:
Report these checks:
- Old path content hit count
- Old path filename/directory-name hit count
- Whether config.toml still contains old paths
- Current marketplace personal source value
- Current automations cwd values
- Whether cap_sid still contains old paths
- Whether global state parses as JSON
- projectless-thread-ids count
- pinned-thread-ids count
- rollout BOM count
- rollout first-event anomaly count
- SQLite old path hit count
- Whether sandbox/capability has been regenerated or is in a regeneratable state

Finally report:
1. Items repaired.
2. Historical text leftovers that were not repaired but have no behavioral impact.
3. A clear instruction that I need to restart Codex Desktop.
```
